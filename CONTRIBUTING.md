# Contributing to Awesome Loop Models

Thank you for contributing.

This repository curates:
- papers on loop models — architectures where, within a single forward process, a shared learned internal layer, block, module, or operator is reused
- selected long-form technical blogs about loop models

---

## Preferred Workflow: Use the Submission Guide First

Please start from the submission guide page:

- submit.html

The guide is the intended staging page before opening a pull request:
- it lets you choose Paper vs Blog explicitly
- it lets you fill the main submission fields once on the guide page
- it exposes searchable existing-tag selectors before you leave for GitHub
- it generates the target YAML path and the generated YAML content
- it tells you exactly what path and YAML to copy into your own fork

GitHub cannot auto-create a fork with the generated YAML already inserted on your behalf from a public link. After reviewing the generated file on the guide page:

1. Fork the repo to your own GitHub account.
2. Create a branch in your fork.
3. Create the generated file path in that branch and paste the generated YAML.
4. Open a pull request back to this repo.

If you already have a fork, skip straight to the branch / file / PR steps. The repository PR template will appear on the final PR page.

Do not edit `README.md`, `papers.json`, or `TAGS.md` by hand. They are generated by `scripts/build.py`.

Before submitting paper metadata, run the read-only catalog audit from the
repository root:

```bash
python3 scripts/audit_catalog.py
python3 scripts/audit_catalog.py --format json
python3 -m unittest tests.test_audit_catalog
```

Errors are blocking structural, identity, URL, or controlled-tag problems;
warnings require human review but do not make the command fail. The auditor
never edits YAML and does not replace primary-source review of paper scope,
descriptions, categories, or semantic tag accuracy. See
[`audits/README.md`](audits/README.md) for the full contract.

## Quick Scope Check

### Papers

A paper belongs here only if:

> Within one model forward process, a shared learned internal layer, block, module, or operator is reused.

Good fits:
- recurrent-depth / looped transformers
- adaptive halting over repeated block application
- deep equilibrium / fixed-point / implicit-layer models
- shared processors repeatedly executed for algorithmic reasoning

Usually out of scope:
- agent loops
- plain autoregressive decoding
- latent reasoning via repeated full forward calls
- diffusion / energy-based reasoning
- recursive model calls
- untied depth scaling
- external search or planning over arbitrary layer orders when the qualifying loop is not a shared internal forward-process recurrence

Standard sequence-time recurrence alone is not enough; the repo's target is iterative depth or latent recurrence inside one forward pass.

### What counts as a blog?

A blog belongs here only if it is a substantive public long-form technical post about loop models or closely related loop-model questions.

Good fits:
- technical deep-dives
- lab blog posts
- long-form essays
- tutorials with real technical content
- X articles / Substack posts / personal blog posts with substantial analysis

Usually out of scope:
- short tweets or short announcement posts
- marketing pages
- bare link roundups with no real analysis
- generic news reposts
- project pages that do not contain meaningful technical writing

Blogs are taggable, but they do not use the paper taxonomy. They live in one flat `Blogs` section.

If a paper or blog is borderline, explain the fit clearly in the pull request description.

See [TAXONOMY.md](TAXONOMY.md) for the full paper taxonomy and the blog-section rule.

## Current Paper Taxonomy

### Primary categories

- `analysis` → `Theoretical and Mechanical Analysis`
- `designs` → `Architecture and Algorithm Designs`
- `applications` → `Applications Focused`

These shelves are flat. New canonical paper YAML should omit `category_path`.

> The paper categories are intentionally coarse. Foundation status plus Loop Mechanism / focus / domain tags carry secondary structure without introducing a separate lineage-tag axis.

If none of these paper categories fit, stop the automatic PR flow and discuss a taxonomy change with the maintainer instead of forcing a bad match.

### How to choose a category

- Use `category: analysis` when the main reader takeaway is theory, mechanism analysis, probing, diagnostics, or formal understanding.
- Use `category: designs` when the main reader takeaway is a new architecture, algorithm, efficiency method, memory-compression method, or training/inference method.
- Use `category: applications` when the main reader takeaway is loop-model performance in a concrete external domain or task, such as robotics, VLA, multimodal, tabular data, graph data, or other domain-specific settings.

### Foundation badge

`foundation: true` is optional secondary metadata for canonical anchor papers such as ACT, Universal Transformers, or DEQ. It is not a separate top-level category anymore.

## Metadata Reference

If you want to help maintainers by drafting metadata before opening GitHub's editor, use one of the following schemas.

### Paper YAML sketch

```yaml
title: "Paper Title"
authors:
  - "First Author"
  - "Second Author"
year: 2026
published_date: "2026-04-20"
added_date: "2026-04-20"
venue: "arXiv"
category: "designs"
foundation: false
mechanism_tags:
  - "flat-loop"
domain_tags:
  - "language-modeling"
tags:
  - "LoopLM"
focus_tags:
  - "architecture"
  - "inference-algorithm"
desc: "One concise sentence describing the paper's core contribution."
links:
  arxiv: "https://arxiv.org/abs/XXXX.XXXXX"
  paper: ""
  github: "https://github.com/..."
  hf: ""
  openreview: ""
  project: ""
  comment:
    - label: "Deep-dive thread"
      url: "https://x.com/..."
  twitter: ""
```

### Blog YAML sketch

```yaml
title: "Blog Title"
authors:
  - "Author Name"
year: 2026
published_date: "2026-04-21"
added_date: "2026-04-21"
venue: "X Article"
mechanism_tags:
  - "flat-loop"
domain_tags:
  - "language-modeling"
tags:
  - "depth-scaling"
focus_tags:
  - "architecture"
desc: "One concise sentence describing the post's technical value."
links:
  blog: "https://x.com/..."
  github: ""
  project: ""
  twitter: ""
  comment:
    - label: "Follow-up discussion"
      url: "https://x.com/..."
```

Notes for papers:
- use `category: analysis` when the main reader takeaway is analytical understanding
- use `category: designs` when the main reader takeaway is a new architecture, algorithm, efficiency method, or memory-compression method
- use `category: applications` when the main reader takeaway is task/domain performance in a concrete external setting
- omit `category_path` in new canonical paper YAML
- use `foundation: true` only for canonical anchor papers; do not use it just to mean "important"
- if a shelf choice is ambiguous, follow the tie-break rules in [TAXONOMY.md](TAXONOMY.md)

Notes for blogs:
- blogs do not use `category`, `category_path`, or `foundation`
- `links.blog` is the canonical primary URL for a blog entry
- use `venue` to describe the source, for example `X Article`, `Lab Blog`, `Substack`, or `Personal Blog`

Notes for both papers and blogs:
- `published_date` is required for all papers and blogs; use the original release date and keep it as a quoted `YYYY-MM-DD` string
- `added_date` is the repo intake date used by the daily feed / Today-style filtering; when included, keep it as a quoted `YYYY-MM-DD` string

### Filename Rules

Use the generated path from `submit.html` instead of inventing a new filename by hand.

- Papers should use `papers/<arxiv_id>.yaml` when the primary link is an arXiv URL.
- Papers without an arXiv ID should use `papers/<shortname>.yaml`, where `shortname` is a stable kebab-case slug.
- Blogs should use `blogs/YYYY-MM-DD-shortname.yaml`.
- For blogs, `YYYY-MM-DD` must match the blog's `published_date`.
- For blogs, `shortname` can be provided manually; if omitted, the submission guide will derive it from the title.
- Blog filenames must not collide with an existing file in `blogs/`.

### Browser-facing tag system

The interactive browser uses three visible tag-filter groups:
- Loop Mechanism (`mechanism_tags`)
- `focus_tags`
- `domain_tags`

Use them as follows:

#### Loop Mechanism (`mechanism_tags`)

Use Loop Mechanism (`mechanism_tags`) only for the loop form. This is a closed vocabulary with exactly four values:

- `hierarchical-loop` for nested, recursive, multi-level, or coarse-to-fine loop structure
- `flat-loop` for a single shared block/layer/module repeated in a flat depth/time loop
- `parallel-loop` for parallel, branched, or multi-path loop execution
- `implicit-layer` for fixed-point, equilibrium, or implicit-layer computation

Do not use abbreviations such as `DEQ`, `UT`, `HRM`, or `Ouro`, and do not use fine-grained labels such as `recurrent-depth`, `adaptive-compute`, `algorithmic-loop`, `memory-compression`, or `recursive-loop` as browser-facing Loop Mechanism tags. Keep paper/model aliases in optional `tags` if useful.

#### Valid `focus_tags`

- `objective-loss`
- `training-algorithm`
- `architecture`
- `data`
- `inference-algorithm`

#### Example `domain_tags`

Use these for the browser-facing domain axis; they are open-ended examples rather than a closed list:
- `language-modeling`
- `reasoning`
- `algorithmic-reasoning`
- `vision`
- `robotics-vla`
- `multimodal`
- `tabular-data`
- `graph-data`

#### Notes on `tags`

`tags` are optional aliases or short model/post identifiers in YAML / README metadata. The interactive browser surfaces Loop Mechanism (`mechanism_tags`), `focus_tags`, and `domain_tags` as filter chips; alias-style `tags` stay as metadata.

See [TAGS.md](TAGS.md) for the current repo-wide tag inventory. Prefer reusing an existing tag and spelling before proposing a new one.

Loop Mechanism and `focus_tags` are controlled vocabularies. If your submission needs a taxonomy or controlled tag-vocabulary change, mention the proposal explicitly in the pull request description and briefly explain why it should be added.

If you want searchable existing-tag selectors before opening GitHub, use the submission guide: submit.html

## Static Site Asset Budgets

After regenerating the canonical site artifacts, run the deterministic, offline budget check:

```bash
python3 scripts/build.py
python3 scripts/check_asset_budgets.py
```

The checker reports every measured value alongside its limit and lists all violations in one run. Current budgets are:

| Asset or schema contract | Limit |
|---|---:|
| `papers.json` raw size | 260,000 bytes |
| `papers.json` deterministic gzip size (`mtime=0`) | 60,000 bytes |
| `papers.json.briefings` | At most 1 item |
| `content` keys anywhere within browser briefings | 0 |
| `submission-meta.json` raw size | 20,000 bytes |
| `assets/favicon.png` raw size | 10,000 bytes |

Missing files, malformed JSON, and invalid generated JSON schemas also fail the check. On every pull request, CI regenerates the offline artifacts, checks these budgets, and runs the unit tests. Citation metrics fetching and generated-file commits remain disabled for pull requests, so the PR gate performs no metrics network access and does not write back to the branch. Do not evaluate the checker against stale generated files locally.

If a budget must change intentionally, first confirm that payload or image reduction is not practical. Then explain the measured change and rationale in the pull request, and update the checker constant, its contract tests, and this table together. Budget increases require maintainer review; do not raise a limit only to make CI green.

## Writing Good Entries

Please keep entries:
- accurate
- conservative about scope
- concise
- English-only

For `desc`, prefer third-person present tense:
- "Introduces ..."
- "Shows ..."
- "Analyzes ..."
