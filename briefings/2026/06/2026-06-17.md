# Daily Loop-Model Briefing — 2026-06-17

## Outcome

Three high-confidence papers were added to the collection, and one existing entry was updated with a reader-facing announcement thread. The additions strengthen today's looped-Transformer coverage from three angles: adaptive fixed-point halting, residual-scaling theory, and practical loop-count selection for large coder models.

## Added to the collection

### Fixed-Point Reasoners: Stable and Adaptive Deep Looped Transformers

Fixed-Point Reasoners addresses a core problem for looped reasoning models: increasing effective depth can improve compositional reasoning, but long recurrent unrolls can destabilize signal propagation and make halting difficult. The paper proposes FPRM, a Transformer-based Fixed-Point Reasoning Model, using pre-norm layers and residual scaling to support stable looped computation.

Why it was added: the abstract explicitly describes a looped architecture and uses fixed-point convergence as an end-to-end halting mechanism. This is same-forward reuse of a learned Transformer reasoning block, not repeated full-model calls or generic iterative prompting.

Main reader takeaway: FPRM treats test-time compute as an adaptive loop depth selected by convergence. The paper reports effectiveness on Sudoku, Maze, state-tracking, and ARC-AGI benchmarks, making it a direct architecture/design addition for looped reasoning.

### On the Residual Scaling of Looped Transformers: Stability and Transferability

This analysis paper studies a basic stability issue in looped weight-tied Transformers. Ordinary residual-depth scaling is not enough when the same residual block is reused many times, because updates are correlated across loop iterations. The paper derives stronger loop-aware scaling, including a factored parameterization for multi-layer blocks.

Why it was added: the abstract explicitly defines looped Transformers as applying the same residual block repeatedly, then analyzes the consequences of that weight sharing. This is directly about the mechanics of same-forward reused learned blocks.

Main reader takeaway: the proposed 1/N loop scaling gives a practical recipe for stabilizing larger recurrent depths and transferring hyperparameters from smaller loop counts without retuning.

### LoopCoder-v2: Only Loop Once for Efficient Test-Time Computation Scaling

LoopCoder-v2 studies Parallel Loop Transformers for code generation. It trains 7B coder models with different loop counts and analyzes the trade-off between extra latent refinement and the positional mismatch introduced across loop boundaries.

Why it was added: the paper is explicitly about looped Transformers that repeatedly apply shared blocks, with Parallel Loop Transformers making the loop count a central architectural and test-time compute choice.

Main reader takeaway: more looping is not automatically better. The two-loop variant delivers the main productive refinement, while three or more loops regress, giving the collection a useful large-model counterpoint to papers that treat additional loop depth as monotonic scaling.
